# M2.5 — Demo Readiness & Documentation Layer — 2026-07-17

## What was implemented

A fresh deployment can now be made demo-ready with a single command. `pnpm
demo:setup` runs migrations and guarantees a permanent demo tenant —
**Meridian Financial Group** (slug `meridian-financial-group`) — a permanent
demo account (**demo@svalbard.ca** / `Demo123`, active admin membership), and
the full deterministic Meridian SOC dataset, then verifies integrity and
prints a summary. A prospect can sign in immediately and explore the platform
with no registration, org creation, or manual seeding.

Demo data is controlled by a new **required** environment variable,
`ENABLE_DEMO_DATA` (`true`/`false`, no implicit default, not derived from
`NODE_ENV`). When `false`, both `pnpm demo:setup` and the in-app seed endpoint
(`POST /api/v1/organizations/current/seed`) refuse — no Meridian data can ever
reach a customer deployment.

This milestone also establishes the mandatory status-documentation process:
every completed task gets a document in `docs/status/` (template:
`docs/status/TEMPLATE.md`).

## Architectural decisions

- **`ENABLE_DEMO_DATA` is required with no implicit default** and is not
  derived from `NODE_ENV`, because the public demo host and future customer
  deployments both run production builds. The behavior must be explicit and
  environment-controlled. The app fails fast at boot if the variable is
  missing.
- **Seed-if-missing, `--reset` to restore.** Re-running `demo:setup` preserves
  modifications made during sales demos (triage, promotions). `--reset`
  re-runs the deterministic wipe-and-rewrite seed, restoring the pristine
  dataset with identical stable IDs.
- **Seed marker in `organization.settings.demoSeed`** (`version`, `seededAt`,
  full `SeedSummary`) instead of a new table/migration. The stored summary is
  also the baseline for integrity verification: fresh seeds must match
  exactly; existing tenants must have at least the seeded counts (demo
  sessions can add incidents but cannot delete seeded entities).
- **Bootstrap does not reuse `createOrganization`**, whose slug logic appends
  `-2`, `-3`… on collision. The demo org is adopted/created by exact slug —
  no suffixes, no duplicates (guarded by `onConflictDoNothing` on the unique
  slug index for concurrent runs).
- **No special-case authentication.** The demo user's password is hashed with
  the same argon2id flow as registration; login goes through the normal
  `/api/login` route. Bootstrap never overwrites an existing password.
- **Active-org pinning at login is generic**, not demo-specific: `/api/login`
  now persists the same default organization that `getOrgContext()` already
  resolved as a fallback. Behavior is unchanged for multi-org users (pinned
  org wins); the demo account lands directly in Meridian.
- **Membership healing:** bootstrap restores the demo membership to
  active/admin if it drifted, since a suspended or downgraded demo account
  would break the demo promise.

## Deviations from plan

- The original plan proposed deriving the `ENABLE_DEMO_DATA` default from
  `NODE_ENV`; per review, the variable is now required with no default.
- Added active-org pinning at login (approved requirement #6), which was not
  in the original plan.

## Open risks

- `Demo123` is below the 10-character minimum enforced at self-registration.
  Accepted deliberately for demo friction reasons; bootstrap writes the hash
  directly so no policy code was weakened.
- The demo org is identified by slug. If an operator manually created an org
  with slug `meridian-financial-group` before running bootstrap, that org is
  adopted as the demo org (acceptable: demo hosts are controlled
  environments; customer hosts run with `ENABLE_DEMO_DATA=false`).
- Integrity verification uses count baselines; it detects missing seeded data
  but not field-level tampering. `--reset` remains the recovery path.
